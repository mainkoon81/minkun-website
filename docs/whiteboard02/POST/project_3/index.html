Product Requirements Document (PRD)

[A]. Overview

(1) background

The ongoing transition toward a climate-neutral energy system is accelerating the integration of decentralized renewable energy sources - often referred to as sector coupling - such as photovoltaic (PV) systems and wind turbines, along with the electrification of transportation through electric vehicles (EVs). Consequently, as described in Figure 01, current low-voltage (LV) grids are experiencing increasing operational stress due to bi-directional power flows and rising peak demand. In this evolving environment, distribution system operators (DSOs) are tasked with ensuring secure and reliable grid operation. Real-time monitoring of the grid state, combined with accurate assessment of peak load (energy consumption) and peak generation (energy injection), is therefore essential to prevent thermal overloading, mitigate voltage violations, and maintain system stability

(2) objective

Smart secondary substations such as digiONS provide real-time feeder cable monitoring, which significantly enhances system observability. However, substation-level monitoring systems with digiONS, particularly those based on SCADA infrastructure, are susceptible to measurement noise, incomplete sampling intervals, aggregation artifacts, and communication failures. These issues reduce the reliability of the data and make direct analytical use difficult. Consequently, systematic data acquisition control and quality assurance mechanisms are indispensable for maintaining the credibility of the LV grid analysis.

  • Goal: Our goal is to ensure the quality of data collected during the operation of the smart secondary substations. We design and implement a scalable data pipeline for operational substation monitoring data using a Medallion architecture (Bronze → Silver → Gold). The proposed pipeline is designed to:
    • ingest operational measurements from substation monitoring systems
    • perform data quality validation
    • transform raw data into analysis-ready datasets
    • provide monitoring dashboards and automated alerting mechanisms.

[B]. Data:

The empirical data used in this study are sourced from the publicly available Kaggle dataset: Power Exchange Data Germany. The dataset provides detailed operational measurements for one representative substation. The investigated substation supplies a LV grid consisting of 89 households, 450 kW of installed PV capacity, and a battery storage system rated at 300 kW / 850 kWh. The operational dataset collected from this substation contains active power measurements recorded at the LV busbar. These measurements are particularly useful for system analysis, as they reflect the combined effects of household demand, photovoltaic generation injected into the grid, and battery operation under bidirectional power flow conditions. In this chapter, these active power measurements serve as the primary basis for investigating data acquisition consistency and quality assurance aspects.

A snapshot of the dataset is shown in Figure 02. Active power value is reported as 15-minute average values (kW), computed from underlying 1-minute measurements. In addition to the average value, the dataset includes the corresponding minimum and maximum 1-minute readings within each aggregation interval, as well as the number of valid samples collected from each 15-minute interval in the SCADA-based aggregation process. However, as discussed earlier, the dataset may contain data quality issues commonly associated with SCADA monitoring systems, such as incomplete sampling intervals, aggregation artifacts, and communication failures. The Available Data Count column shown in Figure 02. reflects this aspect by indicating the number of available measurements used in the aggregation of each interval. To examine these data quality issues, this study formulates eight assurance standards at the level of each 15-minute observation interval:

  • Missing timestamp: each record is checked for missing Timestamp value, which may arise from communication failures or related disruptions.
  • Missing available data count: each record is checked for the presence of the Available Data Count value, which may arise from communication failures or related disruptions.
  • Missing signal identifier: each record is checked for a valid signal identifier so that observations can be uniquely referenced.
  • Duplicate signal identifier: each record is checked to ensure that no duplicated signal identifier exists, so that observations can be uniquely referenced.
  • Timestamp misalignment: each Timestamp value is checked to confirm that it falls on a quarter-hour boundary (00, 15, 30, or 45 minutes), as expected for a stable 15-minute signal feed.

In addition to these structural issues, this study also considers consistency-related problems in the reported active power measurement statistics.

  • Invalid minimum-average relationship: the reported Active Power min value must not exceed the Active Power AVG value, in accordance with the expected ordering of aggregated statistics.
  • Invalid average-maximum relationship: the reported Active Power AVG value must not exceed the Active Power max value, since such a case would indicate inconsistency in the aggregated statistics.
  • Unexpected sample count: the aggregation window is checked for completeness by requiring the Available Data Count measurements within each 15-minute interval to equal 15.

Among these, unexpected sample count is particularly important, since deviations from this expected count may indicate irregularities in the SCADA-based acquisition process. Taken together, these quality issues provide a practical basis for judging whether a recorded interval is sufficiently reliable for subsequent analysis.

[D]. Functional Requirements

  • Medallion Layer Implementation

    • Bronze: Raw CSV data are ingested using Apache Spark (append-only) + metadata (ingestion timestamp, batch identifiers, etc.)

    • Silver: Schema standardization (parsing timestamp, normalizing event types, applying conditional column rules, etc.)

      • validation rules:
        • active_power_min ≤ active_power_mean ≤ active_power_max
        • data_count ≤ 15
        • timestamp continuity check.

    • Gold: Curating data marts designed for analytical use

      • Data Mart for “Load Ratio Analysis”
      • Monitoring Table I. (daily KPI)
        • NO.valid intervals | NO.empty intervals | NO.partial intervals | NO.aligned intervals | completeness rate |
      • Monitoring Table II. (daily issue-type summary)
        • issue type | issue count | first issue ts | last issue ts |
      • Analytical Table
        • signal_id | signal_date | quarter_hour_slot | DataCNT | AVGsig | MINsig | MAXsig |
      • Derived Indicators
        • daily peak load
        • load variability metrics
        • power fluctuation statistics

  • Orchestration (Airflow)

    • Airflow orchestrates the transformation workflow within the Medallion architecture.
      • Schedule execution ⇒ Ingestion using Spark ⇒ dbt run(Silver/Gold) ⇒ dbt test ⇒ slack
      • In case of failure in any transformation or validation, the workflow automatically retries tasks and provides log links for troubleshooting.
      • Data Quality Monitoring (dbt test)
        • not_null, unique, accepted_values, relationships

  • Alerting (Slack)

    • Airflow ensures that each transformation stage is executed sequentially and that downstream tasks are only triggered when upstream validation checks succeed.
      • If DQ-tests fail, automated alerts are sent via Slack notifications.
      • The alerts include information (batch ID, affected tables, causes of the issue, etc.)
      • Rapid detection and resolution of data quality issues within the pipeline.

  • Scheduling

    • The transformation workflow is scheduled to execute on a daily basis to ensure that analytical datasets in the Gold layer remain consistent with the latest available operational measurements. Failure handling mechanisms include:
      • automatic retry attempts
      • task dependency validation
      • alert notifications to system operators.

[E]. Non-functional Requirements

  • SLA (service level agreement)
    • Monitoring and reporting must be available daily.
    • Gold Mart tables must be published before 7 am. with the data derived from the previous day’s records.
    • If dbt tests fail, the publication of Gold tables is blocked as a gatekeeping mechanism.
  • Performance & Cost
    • Daily pipeline execution should complete within 30 minutes.
    • Query costs are managed by using Google Cloud BigQuery table partitioning based on the event_date field.
  • Safety
    • Access to Bronze layer raw data should be restricted to authorized users.

The validation rules focus on four main dimensions: - Completeness, Validity, Consistency, Timeliness

####################################################################
# reads Kafka continuously
# writes to GCS as Parquet
# adds partition columns by processing time
# stays alive and prints a useful error if the stream fails
####################################################################

import time
from pyspark.sql import SparkSession
from pyspark.sql.functions import current_timestamp, date_format, col

# -----------------------------
# 1. Basic config
# -----------------------------
APP_NAME = "kafka-consumer"

BOOTSTRAP = "kafka:9092"
TOPIC = "signal_events"
GROUP_ID = "spark_signal_consumer"

GCS_PATH = "gs://test-kafka-mk/digiONS/raw"
GCS_CHECKPOINT = "gs://test-kafka-mk/checkpoints/digiONS"

GCP_KEYFILE = "/opt/spark/secrets/wired-effort-483112-k2-59425ba2002c.json"

# -----------------------------
# 2. SparkSession
# -----------------------------
spark = (
    SparkSession.builder
    .appName(APP_NAME)
    .config("spark.hadoop.fs.gs.impl", "com.google.cloud.hadoop.fs.gcs.GoogleHadoopFileSystem")
    .config("spark.hadoop.fs.AbstractFileSystem.gs.impl", "com.google.cloud.hadoop.fs.gcs.GoogleHadoopFS")
    .config("spark.hadoop.google.cloud.auth.service.account.enable", "true")
    .config("spark.hadoop.google.cloud.auth.service.account.json.keyfile", GCP_KEYFILE)
    .getOrCreate()
)

spark.sparkContext.setLogLevel("INFO")

print("Spark session created")
print("App name:", APP_NAME)
print("Kafka bootstrap:", BOOTSTRAP)
print("Kafka topic:", TOPIC)
print("GCS path:", GCS_PATH)
print("Checkpoint path:", GCS_CHECKPOINT)

# -----------------------------
# 3. Read Kafka stream
# -----------------------------
raw_df = (
    spark.readStream
    .format("kafka")
    .option("kafka.bootstrap.servers", BOOTSTRAP)
    .option("subscribe", TOPIC)
    .option("startingOffsets", "earliest")
    .option("kafka.group.id", GROUP_ID)
    .option("failOnDataLoss", "false")
    .load()
)

# Keep useful Kafka metadata too
value_df = raw_df.selectExpr(
    "CAST(key AS STRING) AS kafka_key",
    "CAST(value AS STRING) AS value",
    "topic",
    "partition",
    "offset",
    "timestamp AS kafka_timestamp"
)

# -----------------------------
# 4. Add processing-time partitions
# -----------------------------
with_ts = (
    value_df
    .withColumn("proc_ts", current_timestamp())
    .withColumn("event_date", date_format(col("proc_ts"), "yyyy-MM-dd"))
    .withColumn("event_hour", date_format(col("proc_ts"), "HH"))
    .withColumn("event_minute", date_format(col("proc_ts"), "mm"))
)

# -----------------------------
# 5. Write stream to GCS
# -----------------------------
query = (
    with_ts.writeStream
    .format("parquet")
    .option("path", GCS_PATH)
    .option("checkpointLocation", GCS_CHECKPOINT)
    .outputMode("append")
    .partitionBy("event_date", "event_hour", "event_minute")
    .queryName("signals_to_gcs_partitioned")
    .start()
)

print("Streaming query started")
print("Query name:", query.name)
print("Query id:", query.id)
print("Query runId:", query.runId)

# -----------------------------
# 6. Keep alive and show status
# -----------------------------
try:
    while query.isActive:
        print("----- stream status -----")
        print(query.status)
        if query.lastProgress is not None:
            print("Last progress:", query.lastProgress)
        else:
            print("Last progress: None yet")
        time.sleep(10)

    # If stream becomes inactive unexpectedly, surface the reason
    if query.exception() is not None:
        raise Exception(query.exception())

except KeyboardInterrupt:
    print("Stopping stream by user request...")
    query.stop()

except Exception as e:
    print("STREAM FAILED")
    print(str(e))
    raise

finally:
    print("Shutting down Spark session...")
    spark.stop()

[D]. From Silver to Gold Medallion

(1) Three Data Marts

In the Gold layer, the validated measurements (data points) from the Silver layer are aggregated and transformed into analysis-ready tables that can be queried directly without additional preprocessing. To implement this layer, three SQL models - fct_signal_clean_15min.sql, fct_signal_issue_daily.sql, fct_signal_quality_daily.sql, and one YAML file - gold.yml are created and placed in the gold folder under the models directory of the my_dbt_project.

  • fct_signal_quality_daily is a daily KPI monitoring table. It summarizes 15 minute signal intervals at the daily level. Under normal conditions, each day contains \(24 × 4 = 96\) expected intervals. The table reports how many of these intervals are valid, empty, partial, and correctly aligned each day, thereby serving as a daily data quality monitoring dashboard. This allows the KPI related to timestamp alignment to be identified quickly on a daily basis. By showing whether 15-minute intervals are recorded at the correct quarter-hour timestamps, it provides a daily indication of whether DigiONS is functioning properly in terms of timestamp alignment.

  • fct_signal_issue_daily is a daily issue-type summary monitoring table. It summarizes row-level signal-quality issues by day. The issue includes (1) Incomplete 15-minute Window, (2) Missing Minute-level Signals, (3) Delayed Ingestion, (4) Other Collection Problem. For a substation monitoring context, this helps quickly see whether the problem is occasional or persistent, whether it clusters in certain days or periods, how long the problematic window lasted each day, etc.

  • fct_signal_clean_15min is the clean analytical table for daily use. This table is not intended for monitoring; instead, it stores one trusted 15-minute substation interval per row after incomplete, null, misaligned, and internally inconsistent records have been removed. It is therefore suited to downstream analyses such as load-curve plotting, comparison of substation behavior over time, and other applications requiring reliable interval data. Accordingly, it preserves the original structure of the Power Exchange dataset while retaining only trusted records.

GERMANY_TZ = pendulum.timezone("Europe/Berlin")

slack_fail_alert = send_slack_webhook_notification(
    slack_webhook_conn_id="slack_webhook",
    text=(
        ":red_circle: Airflow DAG failed.\n"
        "*DAG*: {{ dag.dag_id }}\n"
        "*Task*: {{ ti.task_id }}\n"
        "*Run ID*: {{ run_id }}\n"
        "*Logical Date*: {{ ds }}\n"
        "*Log URL*: {{ ti.log_url }}"
    ),
)

@dag(
    dag_id="dbt_signal_layered",
    start_date=datetime(2026, 3, 1, tzinfo=GERMANY_TZ),
    schedule="30 12 * * *",   # every day at 12:30 Europe/Berlin
    catchup=True,
    max_active_runs=3,
    tags=["dbt", "signal", "substation"],
    on_failure_callback=[slack_fail_alert],
)
def dbt_signal_layered():
    start = EmptyOperator(task_id="start")

    @task(task_id="run_bronze")
    def run_bronze():
        cmd = "cd /usr/app/my_dbt_project && dbt run --select tag:bronze"
        print(cmd)
        subprocess.run(cmd, check=True, shell=True)

    @task(task_id="run_silver")
    def run_silver():
        cmd = "cd /usr/app/my_dbt_project && dbt run --select tag:silver"
        print(cmd)
        subprocess.run(cmd, check=True, shell=True)

    @task(task_id="run_intermediate")
    def run_intermediate():
        cmd = "cd /usr/app/my_dbt_project && dbt run --select tag:intermediate"
        print(cmd)
        subprocess.run(cmd, check=True, shell=True)

    @task(task_id="run_gold")
    def run_gold():
        cmd = "cd /usr/app/my_dbt_project && dbt run --select tag:gold"
        print(cmd)
        subprocess.run(cmd, check=True, shell=True)

    @task(task_id="test_silver")
    def test_silver():
        cmd = "cd /usr/app/my_dbt_project && dbt test --select tag:silver"
        print(cmd)
        subprocess.run(cmd, check=True, shell=True)

    end = EmptyOperator(task_id="end")

    start >> run_bronze() >> run_silver() >> run_intermediate() >> run_gold() >> test_silver() >> end


dag = dbt_signal_layered()
